# Charts オーダーUI: 上下2表（上=確定/下=候補）レイアウト仕様（cmd_20260214_03）

- 作成日: 2026-02-14
- 対象: Webクライアント Charts のオーダー入力（`OrderBundleEditPanel`）
- 目的: 旧来Swing版の「上:確定（セット）/下:候補（検索結果）」の作業感（同時把握 + 高密度 + キーボード短縮）を、Web版の利点（非モーダル運用 + 可変パネル + 安全支援）を損なわずに統合する。
- 制約: 機能変更禁止（状態遷移・API・登録処理・データ内容の振る舞いは変更しない）。

## 0. 用語

- 確定エリア（上段）: いま「確定した内容」を常時視認できる領域。
  - 最低限: 既に登録済みのオーダー束（登録済み一覧）。
  - 任意: 今回入力中のドラフト束（フォーム内容）を "未確定" として差し込む（視認性のため）。
- 候補エリア（下段）: 検索・推薦・マスタ候補など「次に選ぶ候補」を一覧して選択できる領域。

## 1. スコープ

対象エンティティ（主要カテゴリ）:
- 処方（`medOrder`）
- 注射（`injectionOrder`）
- 検査（`testOrder` / `physiologyOrder` / `bacteriaOrder` / `laboTest`）
- 放射線（`radiologyOrder`）
- 処置（`generalOrder` / `treatmentOrder`）
- 算定（`baseChargeOrder` / `instractionChargeOrder`）

共通要件:
- どのカテゴリでも「上=確定/下=候補」の構造と基本キーボード導線を統一する。
- 候補のデータソースはカテゴリにより異なって良い（既存の `OrderEntityUiProfile` / master presets を尊重）。

非スコープ:
- ORCA/禁忌チェックの仕様変更
- スタンプ/セットの保存形式変更
- 登録処理・送信・会計フローの変更

## 2. レイアウト仕様（必須）

### 2.1 縦分割（上下2表）

- `OrderBundleEditPanel` の主要表示領域は、縦に2分割する。
  - 上段: 確定エリア（常時視認）
  - 下段: 候補エリア（検索結果/候補/頻用/マスタ）

最低条件（常時視認の定義）:
- 上段はスクロールで隠れてはならない。
- パネルの高さが極端に小さい場合でも、上段は "ヘッダ + 最低1行"（または "ヘッダ + 要約"）を必ず残す。

推奨の実装モデル:
- 外枠: `display: grid; grid-template-rows: minmax(var(--confirmed-min-h), var(--confirmed-h)) 1fr;`
- 上段/下段ともに内部スクロール (`overflow: auto`) を許可し、外枠は極力スクロールしない。

### 2.2 固定領域とスクロール

- 上段（確定）:
  - ヘッダは上段内で `position: sticky; top: 0` とし、上段の内部スクロール時も見失わない。
  - 行が多い場合は上段内でスクロール。
- 下段（候補）:
  - 候補リストは下段内でスクロール。
  - 検索条件UI（プリセット切替/キーワード等）は下段内の上部に固定（sticky）して良い。

禁止:
- ページ全体スクロールに依存して上段が消えること。
- 候補の描画が重く、スクロールが引っかかる構造（無限に長いDOMを積む等）。

### 2.3 可変サイズ（リサイズ追随）

前提:
- Charts のユーティリティパネル自体がリサイズ可能（既存仕様）。

追加要件（上下2表の内側）:
- 上下境界に "スプリッタ" を置き、上段高さをユーザーが調整できる。
- 上段の高さは以下の制約を守る。
  - `min`: 56px（ヘッダ + 要約）または 120px（ヘッダ + 1行）を下限
  - `max`: パネル高さの 70% 程度（候補が常に残る）

保存（推奨）:
- 上段高さのユーザー設定を localStorage に保存し、ユーザー/施設/端末で復元する。
  - 例: `opendolphin:web-client:charts:order-two-table-layout:v1:<facilityId>:<userId>:<entity>`

## 3. 情報密度（列設計）

### 3.1 確定表（上段）: 表示項目

目的:
- "今、何が確定しているか" を1秒で把握する。

必須列（最小セット）:
- 状態: `保存済` / `未確定(入力中)` / `送信待ち` など（既存データで表現できる範囲）
- 名称: 束名（bundleName）と代表項目名（先頭アイテム等）
- 指示: 用法/投与指示/検査指示（entityごとの instruction）
- 主要項目サマリ: 件数 + 主要コード/数量
- 安全バッジ: 禁忌/警告/注意、missingMaster/fallbackUsed のバッジ

推奨列（余裕があれば）:
- 開始日/日付
- コピー/編集/削除アクション（既存の操作を維持）

### 3.2 候補表（下段）: 表示項目

目的:
- 候補の "比較" と "確定への移送" を高速化する。

必須列（最小セット）:
- ソース: 施設傾向/患者傾向/ORCAマスタ/補正候補/コメント候補 等
- コード（あれば）
- 名称
- カテゴリ（あれば）
- 安全アイコン: 禁忌/警告/一般名 等（表示可能なものは必ず出す）

推奨:
- 注記/備考（短く）
- マスタ種別ラベル（generic-class / etensu / youhou / material / kensa-sort 等）

省略してよい（ただし必要時に辿れる）:
- 長文の詳細（ポップオーバー/詳細パネルに退避）

## 4. キーボード導線（必須）

目的:
- "マウス無し" でも、候補選択→確定（入力反映）→次候補へ、が詰まらない。

基本方針:
- 表はロービングtabindex（または `aria-activedescendant`）で "選択行" を持ち、上下キーで移動。
- `Enter` は "候補を確定側へ反映"（現在選択中の入力行へ適用、または束へ追加）。
- `Esc` は候補選択を解除し、直前の入力欄へ戻す。

必須ショートカット（提案）:
- `F6`: フォーカスを循環（確定表 → 候補表 → フォーム主要入力 → 確定表 ...）
- `Ctrl+Enter`: 「展開する」（既存の主要アクションに割当）
- `Ctrl+Shift+Enter`: 「展開継続する」

必須のフォーカス可視化:
- フォーカスリング/選択行ハイライトを明確にする。
- フォーカス移動時は aria-live を乱発しない（必要最小限）。

## 5. 安全支援（欠落させない要件）

原則:
- "確定" と "候補" のどちらの文脈でも、安全情報が見えない状態を作らない。

必須:
- 禁忌チェックの結果（警告/エラー/続行可否）を、確定エリアのヘッダ付近に常時表示できる。
- missingMaster/fallbackUsed/dataSourceTransition 等の状態は、確定エリアにも表示し、候補だけに閉じ込めない。
- 一般名表示（処方等で既存表示がある場合）は維持する。

推奨:
- 候補表の各行にも "警告の芽"（例: 禁忌候補、注意薬、コメント帯の注意）をアイコンで示し、詳細はホバー/クリックで説明する。
- 色は `docs/web-client/ux/charts-semantic-color-system-20260212.md` のトーンに合わせる。

## 6. 実装メモ（差分最小での落とし込み）

狙い:
- 既存ロジック（検索/推薦/禁忌/登録）を極力触らず、"見せ方" と "フォーカス制御" の変更に留める。

実装方針（案）:
- `OrderBundleEditPanel.tsx` を "表示レイアウト" と "データ/操作" に分離する。
  - `OrderTwoTableLayout`（新規）: 上下分割 + スプリッタ + フォーカスマネージャ
  - `OrderConfirmedTable`（新規）: 既存の「登録済み一覧」UIを表形式へ寄せる（まずは現状のリストを上段へ移動でも良い）
  - `OrderCandidateTable`（新規）: 既存の候補表示（頻用候補/予測候補/コメント候補等）を "1つの表" に正規化して出す

候補の正規化（例）:
- `recommendationCandidates`（頻用オーダー候補）
- `itemPredictiveCandidates`（入力候補マスタ）
- `selectionCommentCandidates`（選択式コメント候補）
- （必要に応じて）マスタ検索結果、コード補正候補

適用先（どこへ反映するか）:
- "現在選択中の入力行"（`selectedItemRowId`）に候補を適用する。
- 適用後のフォーカス:
  - 既定: 候補表に残す（連続選択）
  - `Enter` と `Shift+Enter` で "残す/戻す" を切替できると高速。

非機能:
- 候補表は仮想化（必要なら）を検討。ただし過度な導入は避け、まずはDOM量の上限を管理する。
- 監査ログ/telemetry のイベントは "新規導線" で追加されても良いが、既存イベントの意味を壊さない。

## 7. テスト観点（足軽向けメモ）

ユニット/局所（例）:
- 上段がスクロールで隠れない（DOM構造とCSSクラスの検証）
- `F6` で確定/候補/フォームのフォーカス遷移が循環する
- `Enter` で候補が選択行へ適用される（selectedItemRowId がある場合）
- 禁忌/警告バッジが確定エリアにも表示される

E2E（家老担当）:
- 各カテゴリで "候補→確定" の作業感が再現される（処方/注射/検査/放射線/処置/算定）
- リサイズで崩れない（狭い幅/低い高さ/最大化）
- 「機能変更なし」を説明できる（API/状態遷移/登録結果が一致）
