# 管理画面: マスタ更新コンソールUX案（RUN_ID=20260212T003351Z）

- 更新日: 2026-02-12
- 対象: `web-client` Administration（system_admin）
- 関連設計: `docs/server-modernization/orca-master-reference-update-platform-design-20260212.md`

## 1. 目的
- 更新運用が迷子にならないことを最優先にする。
- ORCAマスタと外部マスタを、同じ運用体験で自動/手動更新できるようにする。
- 失敗時の復旧（再試行・ロールバック）をUIから完結させる。

## 2. 画面構成（最小）
- 更新ダッシュボード
- データセット詳細
- スケジュール設定

## 3. 画面ワイヤー

### 3.1 更新ダッシュボード

```text

+----------------------------------------------------------------------------------+

+----+----------------------+------------------+-----------+------------+-----------+
|状態| データセット         | 最終更新         | 現行件数  | 更新検知   | 操作      |
+----+----------------------+------------------+-----------+------------+-----------+
|正常| ORCA core master     | 2026-02-12 08:30 | 148,221   | 更新なし   | [詳細]    |
|失敗| PMDA insert index    | 2026-02-12 03:00 |  95,012   | 更新あり   | [詳細]    |
```

表示要素:
- 状態: `正常/失敗/更新あり/停止`
- 更新検知: ORCAは `masterlastupdatev3` の比較結果を表示
- 失敗要約: 1行で原因、時刻、再試行回数を表示

## 4. 手動操作の安全設計
- 二重起動防止: `実行中` のとき同一datasetのボタンを無効化（API側も lock で拒否）。
- 監査ログ: `actorId`、`action`、`dataset`、`version`、`timestamp`、`result` を記録。
- 確認UI:
  - ロールバック: `現在版` と `切戻し先` の件数差分を表示。
  - アップロード: `形式OK/件数妥当性/ハッシュ重複` を明示。

## 5. オーダー入力連携の見せ方
- 管理画面では「診療中参照DBのみ利用中」をバッジ表示し、外部直参照していないことを明示。
- オーダー画面側には参照DBの `asOf` を表示し、更新遅延時は注意喚起する。
- 最終算定は ORCA結果優先であることを共通メッセージとして統一。

## 6. エラー文言方針（例）
- `外部取得に失敗しました (PMDA, HTTP 503)。現行版を継続利用しています。`
- `形式エラー: 必須列 package_unit が不足しています。テンプレートを確認して再アップロードしてください。`
- `ロールバック完了: ORCA core master を 2026-02-11 03:00 版へ切戻しました。`

## 7. 受け入れ条件（UI）
- ダッシュボードで全datasetの状態・最終更新・件数・失敗要約が1画面で分かる。
- 詳細画面で手動更新/アップロード/ロールバックが可能。
- スケジュール設定で時刻・再試行・timeout・同時実行上限・ORCAポーリング間隔を設定可能。
- 実行中の二重起動がUI/APIの両方で防止される。
