# server-modernized 受け入れ条件（P1-02）

- 更新日: 2026-03-11
- RUN_ID: 20260310T204637Z
- 目的: `docs/modernization/business-critical-flows.md` で定義した最重要業務フロー（CF-01〜CF-07）の合否判定を固定する。
- 判定原則: UI や旧 API 形状ではなく、業務結果（登録・保存・連携・取得）の成否で判定する。

## 1. 共通テンプレート
各フローは次の4項目を必須とする。

| 項目 | 記載内容 |
|---|---|
| 入力 | 前提データ、操作、リクエスト、外部条件 |
| 期待結果 | 業務上成立しているべき状態（保存・反映・応答） |
| 確認項目 | DB/ログ/応答/取得結果などの検証観点 |
| 失敗時の扱い | ロールバック、再試行、利用者通知、監査記録 |

## 2. 受け入れ条件一覧

### CF-01 受付登録（優先 S）
- 入力:
  - 当日診療日で受付対象患者（既存患者ID）を指定し、受付登録を1件実行する。
  - 同一患者で同日重複受付の試行を1回含める。
- 期待結果:
  - 新規受付は受付一覧に反映され、受付状態が「受付済み」として参照できる。
  - 重複受付試行は重複として拒否または統合され、業務データ不整合を起こさない。
- 確認項目:
  - 受付データの登録件数と患者紐付けが正しい。
  - 受付状態を取得するAPI/一覧で同一状態が確認できる。
  - 重複試行時に同一患者・同日で重複行が増えていない。
- 失敗時の扱い:
  - 登録途中失敗時は部分保存を残さずロールバックする。
  - 利用者へ再実行可能なエラーを返し、監査/運用ログへ失敗理由を残す。

### CF-02 カルテ保存・参照・改訂（優先 S）
- 入力:
  - 対象患者でカルテ新規作成を行い、保存後に参照、続けて1回改訂を実行する。
- 期待結果:
  - 新規保存文書は患者に紐付いて参照可能である。
  - 改訂後は旧版と新版の関係が保持され、最新版参照と版履歴参照が矛盾しない。
- 確認項目:
  - 文書ID/版番号/患者IDの対応が一意である。
  - 旧版内容が改訂で上書き消失していない。
  - 最新版取得と版指定取得で期待内容を返す。
- 失敗時の扱い:
  - 改訂失敗時は版関係を壊さず、旧版を引き続き参照できる。
  - 失敗応答は再試行可否が判別でき、障害記録に版情報を残す。

### CF-03 ORCA 連携（優先 S）
- 入力:
  - 患者検索、患者更新、受付/会計関連の代表連携をそれぞれ1件以上実行する。
  - 代表失敗系（業務エラーまたはタイムアウト）を1件含める。
- 期待結果:
  - 正常系は業務継続に必要な応答を返す。
  - 失敗系は分類されたエラーとして返却され、呼び出し側が再試行/中断を判断できる。
- 確認項目:
  - ORCA応答の成功/失敗が業務サービスの判定に正しく反映される。
  - タイムアウト時に無限待機せず、設定値どおりに終了する。
  - ログが個人情報を過剰出力せず、追跡に必要な識別子は残る。
- 失敗時の扱い:
  - ORCA失敗で内部データを不整合状態にしない。
  - 失敗分類（通信/業務/認証等）を保持し、運用が切り分け可能である。

### CF-04 添付画像/PDF 保存・取得（優先 S）
- 入力:
  - 患者に対して代表画像1件とPDF1件を保存し、保存後に取得する。
- 期待結果:
  - 添付は患者へ正しく紐付き、取得内容が保存内容と一致する。
  - メタ情報（ファイル種別・サイズ・登録日時）が参照できる。
- 確認項目:
  - バイナリ比較またはハッシュ比較で内容非破壊を確認する。
  - 患者切替時に他患者の添付へ誤参照しない。
  - 取得APIのメタ情報と保存時情報が一致する。
- 失敗時の扱い:
  - 保存失敗時は不完全データを残さない。
  - 取得失敗時は対象識別子をログ化し、再取得可否を判別できるエラーを返す。

### CF-05 患者登録・更新（優先 A）
- 入力:
  - 新規患者登録、既存患者更新、重複候補登録を各1件実行する。
- 期待結果:
  - 新規は一意な患者IDで登録される。
  - 更新は対象患者にのみ反映され、主保険など主要属性の整合性を保つ。
  - 重複候補は重複ルールに従い拒否または明示確認となる。
- 確認項目:
  - 患者ID採番規則と重複判定結果が定義どおりである。
  - 更新前後の差分が対象患者に限定される。
  - 保険情報の主従関係が崩れていない。
- 失敗時の扱い:
  - 失敗時は途中更新を残さずロールバックする。
  - 入力不備と競合（重複/排他）を区別したエラーを返す。

### CF-06 管理設定・認証（優先 A）
- 入力:
  - 管理者で設定変更と保存、一般ユーザーで閲覧、無権限ユーザーで保護操作を試行する。
  - ログイン・ログアウトを1サイクル実行する。
- 期待結果:
  - 管理者は必要設定を保存できる。
  - 無権限ユーザーは保護操作を実行できない。
  - ログアウト後は保護APIへ再認証なしでアクセスできない。
- 確認項目:
  - 権限ごとの応答コード/応答内容が一貫している。
  - 設定変更が永続化され、再取得で一致する。
  - ログイン状態とセッション無効化が期待どおりに動作する。
- 失敗時の扱い:
  - 保存失敗時は旧設定を保持し、中間状態を残さない。
  - 認証/認可エラーは監査ログに操作者と操作対象を記録する。

### CF-07 PVT 受信反映（優先 A）
- 入力:
  - 代表PVTメッセージ3種（通常、重複、異常）を順に投入する。
- 期待結果:
  - 通常メッセージは業務データへ反映される。
  - 重複メッセージは二重登録せず冪等に処理される。
  - 異常メッセージは隔離・失敗記録され、正常処理を阻害しない。
- 確認項目:
  - メッセージ識別子単位で処理結果（成功/重複/失敗）が追跡できる。
  - 再試行後に最終状態が一貫する。
  - 失敗件数と対象IDが運用確認できる。
- 失敗時の扱い:
  - 処理失敗メッセージは再試行可能な形で保持する。
  - 毒メッセージは通常キューから分離し、手動介入可能にする。

## 3. 運用ルール
1. 本書の受け入れ条件は P1-04〜P1-09 の自動テスト設計の入力とし、各テストケースは CF-ID を必ずトレースする。
2. 合否判定で迷った場合は「業務結果が継続できるか」を優先し、UI/旧契約互換の有無を判定基準にしない。
3. 受け入れ条件の変更時は `docs/modernization/business-critical-flows.md` と同時更新し、ID整合性を維持する。
