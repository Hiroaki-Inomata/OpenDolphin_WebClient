# server-modernized 刷新方針（固定版）

- 更新日: 2026-03-11
- RUN_ID: 20260310T203518Z
- 適用範囲: `server-modernized` の設計・実装・試験・運用準備
- 位置付け: 本書は刷新作業の判断基準を固定するための正本。個別タスクの判断が割れる場合は本書を優先する。

## 1. 方針サマリー（固定）
1. 後方互換は前提にしない。旧 API/旧 DTO/旧レスポンス形式を温存しない。
2. 旧入口（Touch/ASP/XML 専用口）は段階的に廃止し、公開契約を JSON に統一する。
3. ORCA 連携は業務ドメインから分離し、境界を明示した adapter 層へ集約する。
4. `common` は責務別モジュールへ分割し、循環参照と直結依存を解消する。
5. ノードローカルファイル保存（`user.home`/`target`/固定パス依存）を廃止し、DB・オブジェクトストレージ・集中ログへ移行する。

## 2. 採否基準（Architecture Decision Gate）
提案・実装・レビューは以下 5 観点で採否判定する。

| 判定観点 | 採用条件 | 不採用条件 |
|---|---|---|
| 互換方針 | 旧互換を増やさず、新契約を明確化している | 旧挙動を延命するための分岐・二重実装を追加する |
| API 公開面 | `/api/v1` + JSON へ収束している | Touch/ASP/XML 専用の入口を新規追加・温存する |
| ORCA 境界 | ORCA 依存が adapter 層に閉じる | Resource/Service へ ORCA 詳細が漏れる |
| モジュール境界 | 依存方向が単方向で循環しない | `common` 集約・相互参照で責務が混在する |
| 永続化/状態管理 | ローカルファイル依存を増やさない | 実行ノード依存の状態保存を追加する |

## 3. 非採用案（記録）
### 3.1 旧 API の併存（長期移行）
- 判定: **非採用**
- 理由: 入口維持コストと検証コストが増大し、刷新完了条件を曖昧化するため。

### 3.2 XML/plist 返却の残置
- 判定: **非採用**
- 理由: JSON 統一方針と矛盾し、DTO/Converter の二重管理を発生させるため。

### 3.3 ORCA 呼び出しのドメイン層直接利用
- 判定: **非採用**
- 理由: 障害切り分け・契約変更追従・試験スタブ化が困難になるため。

### 3.4 `common` 一括維持
- 判定: **非採用**
- 理由: 境界不明瞭・循環依存・保守停滞を継続させるため。

### 3.5 ローカルファイル永続化の継続
- 判定: **非採用**
- 理由: 複数ノード運用・再デプロイ・監査要件に適合しないため。

## 4. 運用ルール
1. 本書に反する実装提案は、WBS 上で代替案を明記しない限り取り込まない。
2. 例外を設ける場合は、期限・撤去条件・影響範囲を同時に記録する。
3. 例外の記録先は `docs/DEVELOPMENT_STATUS.md` と task 別実施記録（`docs/modernization/` または `docs/server-modernization/` 配下）とする。完了済み WBS は archive 参照専用とする。
